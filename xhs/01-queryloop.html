05流循环体 · 50 行做 3 件事

代码里用 ⭐ START / ⭐ END 注释包住的那 50 行 —— 是 harness 跟 stream 协议固化耦合的唯一位置。每条 stream message 进来都过这三道。

① Tombstone

是否要"墓碑"已经 yield 的内容？

fallback 触发过 → 之前累积的 assistant message 全作废 —— 清空 assistantMessages/toolResults/toolUseBlocks，给已经 yield 出去的 tool_use 补 tombstone 让 UI 划掉。

② Clone & Yield

对外那份要 clone 再改

tool_use 块要 backfill 派生字段（如 file path 展开），但只 clone 给"对外 yield 的那份"，原始 message 不动。原因：原始要回灌 API，mutate 会破 byte-level prompt cache。

③ Push & 收集

tool_use 是"再来一轮"的唯一信号

message.type === 'assistant' → push 到 assistantMessages，扫 content 收集 tool_use 块。任何 tool_use 出现就置 needsFollowUp = true —— 这是决策树的入口。

为什么这 50 行最重要：一份 message 在这里被分成 3 个去向 —— ① yield 给消费者（UI 用，可改）、② push 到 assistantMessages（要回灌 API，不可改）、③ 喂给 streamingToolExecutor。每个去向对消息的要求不同，这就是 backfill clone 那套绕弯的根本原因。

06流读完了 · return 还是 continue?

stream 耗尽后，照 needsFollowUp 二分。每个 continue 都是把 state 整体重写一次后跳回 while 顶。
